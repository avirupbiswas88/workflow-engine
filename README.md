# 🤖 High-Throughput Workflow Execution Service

A reactive, non-blocking asynchronous event-driven workflow engine built using Spring Boot 4.0 (Spring Framework 7.x), Project Reactor, and H2 R2DBC. This service is designed to scale horizontally and handle high-frequency requests, targeting a throughput of 10,000+ Transactions Per Second (TPS)

## 📋 Functional & Technical Requirements
### Functional Scope
* **Workflow Definition Persistence:** Store static and dynamic multi-step step configurations in an in-memory H2 store.
* **Sequential Dynamic Execution:** Run steps sequentially, allowing downstream steps to transparently ingest outputs generated by upstream operations.
* **Telemetry Visibility:** Expose a non-blocking streaming progress endpoint detailing the exact execution status and step pointers.
* **Audit Trail History:** Return comprehensive audit records containing step-level execution metrics, timing states, and precise payloads.

### Technical Scope
* **Reactive Non-Blocking Engine:** End-to-end event-driven architecture using Spring WebFlux and R2DBC to guarantee zero thread-starvation under peak load.
* **10,000+ TPS Target Scale:** Compute pipelines are fully decoupled from immediate I/O request-response lifecycles.
* **Resiliency & Fault Tolerance:** Integrated retries with exponential backoffs and timeouts to manage transient downstream network failures cleanly.

## 🏗️ Architecture Overview

The system uses a completely asynchronous, event-driven pattern designed around the Reactive Streams Specification:
```text
                    ┌────────────────────────┐
                    │      REST Client       │
                    └───────────┬────────────┘
                                │ (HTTP POST /execute)
                                ▼
                    ┌────────────────────────┐
                    │   WorkflowController   │
                    └───────────┬────────────┘
                                │ (Non-blocking Dispatch)
                                ▼
                    ┌────────────────────────┐
                    │     WorkflowEngine     │
                    └─────┬────────────┬─────┘
                          │            │
         (Save Initial)   │            │ (Offload to Worker Thread)
                          ▼            ▼
   ┌────────────────────────┐        ┌────────────────────────┐
   │   H2 R2DBC Repository  │        │ BoundedElastic Workers │
   └────────────────────────┘        └───────────┬────────────┘
                                                 │
                                                 │ Loops through Steps
                                                 ▼
                                     ┌────────────────────────┐
                                     │ Step Processors (xN)   │
                                     │  - Circuit Breakers    │
                                     │  - Retry Filters       │
                                     └────────────────────────┘

```

1. **Ingest Phase:** The request enters through WebFlux Netty, avoiding standard thread-per-request limitations.
2. **Persistence Handshake:** An execution shell instance is initialised via an active DatabaseClient transaction.
3. **Compute Decoupling:** The request returns an immediate HTTP 202 Accepted token containing the system UUID. The actual execution pipeline is offloaded to a Schedulers.boundedElastic() thread pool to safeguard the netty runtime loop.
4. **Resilient Sequencing:** Flux.fromIterable(steps).concatMap(...) forces strict sequencing while executing business logic asynchronously, maintaining thread isolation throughout the entire path.

## 🔑 Key Architectural Assumptions

* **Compute Isolation:** Individual workflow steps are executed as high-frequency non-blocking tasks. CPU-intensive operations (such as cryptography, document parsing, or extensive data maps) are assumed to run within dedicated elastic schedulers to protect the main I/O loops.
* **Context Schema Evolution:** Step contexts are serialized into structural CLOB text blobs within the database engine. Downstream actions are assumed to have prior knowledge of the internal key-value schemas produced by upstream steps.
* **Single-Node In-Memory Assumption:** For ultra-high TPS benchmarking, the state engine uses a local, non-replicated H2 reactive database instance.

## ⚖️ Trade-offs Made

### 1. In-Memory H2 Storage vs. Distributed Consensus Stores
* **Trade-off:** Chose an in-memory R2DBC H2 database over a distributed cluster engine (like PostgreSQL, CockroachDB, or Cassandra).
* **Reasoning:** Prioritized raw throughput and zero networking overhead to reliably clear the 10,000 TPS goal on single-node hardware, treating persistence as short-term session state.
### 2. Global Thread-Safe Context Locking vs. Functional State Immutability
* **Trade-off:** Used a concurrent shared context structure (ConcurrentHashMap) across step loops rather than generating fully distinct immutable state instances at every step boundary.
  **Reasoning:** Drastically reduces runtime garbage collection overhead, preventing memory allocation spikes under high load.
### 3. Decoupled Processing vs. Immediate Strong Consistency
* **Trade-off:** Used an Eventual Consistency paradigm where clients receive an immediate 202 Accepted tracking token while calculations process in the background.
* **Reasoning:** A synchronous approach would tie up connection threads while waiting for long-running downstream steps to respond, causing severe resource starvation at high scale.

## 🔮 Future Enhancements

If allocated an additional week of development time to make this a distributed enterprise solution, the following architectural upgrades would be implemented:
* **Distributed State Machine (Redis Backplane):** Replace the single-node H2 memory footprint with an R2DBC Postgres cluster or a Reactive Redis cluster. This allows individual workflow steps to resume execution on separate network cluster nodes if a worker instance crashes.
* **Kafka Event Log Ingestion:** Transition from inside-the-JVM elastic thread schedulers to an enterprise event bus like Apache Kafka or RabbitMQ. Workflows would emit event messages to specific partitions, enabling clean dead-letter-queue (DLQ) support and seamless backpressure management.
* **Distributed Optimistic Locking Tuning:** Enhance database row updates with advanced optimistic conditional step-checking strategies (WHERE version = :currentVersion AND STATUS = 'RUNNING') to guarantee that multi-regional instances never execute duplicate step processing operations simultaneously.
* **Dynamic Custom Engine Registries:** Move past standard hardcoded switch-case blocks by creating an open Reactive Plugin Strategy Engine. Developers can load custom business logic by registering step beans that implement a unified functional execution interface.

## 🛠️ Local Quick Start Guide

Follow these steps to build, run, and test the High-Throughput Workflow Execution Service on your local environment.

### 📋 Prerequisites
Before running the service, ensure your local development environment has:
* **Java Development Kit (JDK):** Version 17 or 11
* **Apache Maven:** Version 3.8 or higher
* **cURL** (or any API testing tool like Postman)

---

### 1. Build and Compile the Application
Clean the target directories and compile the system into an executable JAR package:
```bash
mvn clean package
```

### 2. Launch the Service Runtime
Run the embedded high-performance Netty server locally. The application defaults to port `8080`:
```bash
mvn spring-boot:run
```

---

### 3. Interactive Testing Protocol

#### 🔹 Step A: Submit a Workflow for Asynchronous Execution
Send a payload defining a series of sequential contract processing actions. This endpoint processes data asynchronously and returns a tracking token immediately.

```bash
curl -X POST http://localhost:8080/workflows/execute \
-H "Content-Type: application/json" \
-d '{
  "workflowId": "contract-review",
  "steps": [
    { "name": "validate", "input": { "contractId": "123" }},
    { "name": "approve" },
    { "name": "execute" }
  ]
}'
```

* **Expected HTTP Status:** `202 Accepted`
* **Expected Response Payload:**
  ```json
  {
    "executionId": "b6a84c8a-7e3e-4b68-b733-149fa8619bc9"
  }
  ```

#### 🔹 Step B: Check Live Execution Telemetry (Progress)
Track the active state machine phase and step pointers. Replace the UUID below with the actual `executionId` from Step A:

```bash
curl http://localhost:8080/workflows/executions/b6a84c8a-7e3e-4b68-b733-149fa8619bc9/progress
```

* **Expected HTTP Status:** `200 OK`
* **Expected Response Payload:**
  ```json
  {
    "executionId": "b6a84c8a-7e3e-4b68-b733-149fa8619bc9",
    "status": "RUNNING",
    "currentStep": 2
  }
  ```

#### 🔹 Step C: Retrieve Full Audit Trail (History)
Fetch the consolidated state log showing outputs merged from every completed step execution:

```bash
curl http://localhost:8080/workflows/executions/b6a84c8a-7e3e-4b68-b733-149fa8619bc9/history
```

* **Expected HTTP Status:** `200 OK`
* **Expected Response Payload:**
  ```json
  {
    "executionId": "b6a84c8a-7e3e-4b68-b733-149fa8619bc9",
    "status": "completed",
    "steps": [
      { "name": "validate", "status": "completed", "output": { "valid": true } },
      { "name": "approve", "status": "completed", "output": { "approvedBy": "system" } },
      { "name": "execute", "status": "completed", "output": { "result": "success" } }
    ]
  }
  ```
